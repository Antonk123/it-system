<objective>
Build the complete backend for recurring ticket templates: database tables, CRUD API routes, and the background scheduler that auto-creates tickets on schedule.

Purpose: This is the data foundation for the recurring tickets feature. The frontend (Plan 02) depends on these API endpoints existing. The scheduler is the core automation that fulfills the "auto-create tickets" requirement.

Output: Two new DB tables (recurring_templates, recurring_ticket_history), a full CRUD+history API at /api/recurring, and a cron-based scheduler that fires every minute to process due templates.

Note on RECUR-01 scope: RECUR-01 mentions "anpassad cron" (custom cron expressions) as a scheduling option. Per D-01 locked decision, custom cron syntax is explicitly excluded — simple intervals (daily/weekly/monthly) cover the requirement scope. This is intentional and sufficient for the vast majority of use cases.

**connection.ts — Add ensureRecurringTemplatesTable():**

Add a new function `ensureRecurringTemplatesTable` following the exact `ensureTicketRemindersTable` pattern. It creates TWO tables:

Table 1 — `recurring_templates`:
```sql
CREATE TABLE IF NOT EXISTS recurring_templates (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  title TEXT NOT NULL,
  description TEXT NOT NULL DEFAULT '',
  priority TEXT DEFAULT 'medium' CHECK(priority IN ('low','medium','high','critical')),
  category_id TEXT REFERENCES categories(id) ON DELETE SET NULL,
  tags TEXT DEFAULT '[]',
  interval_type TEXT NOT NULL CHECK(interval_type IN ('daily','weekly','monthly')),
  interval_day INTEGER,
  is_active INTEGER DEFAULT 1,
  last_run TEXT,
  next_run TEXT NOT NULL,
  created_at TEXT DEFAULT CURRENT_TIMESTAMP,
  updated_at TEXT DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX IF NOT EXISTS idx_recurring_templates_next_run ON recurring_templates(is_active, next_run);
```

Table 2 — `recurring_ticket_history`:
```sql
CREATE TABLE IF NOT EXISTS recurring_ticket_history (
  id TEXT PRIMARY KEY,
  template_id TEXT NOT NULL REFERENCES recurring_templates(id) ON DELETE CASCADE,
  ticket_id TEXT NOT NULL REFERENCES tickets(id) ON DELETE CASCADE,
  created_at TEXT DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX IF NOT EXISTS idx_recurring_history_template ON recurring_ticket_history(template_id, created_at DESC);
```

**recurringScheduler.ts — Create new file:**

Create `server/src/lib/recurringScheduler.ts` following the `autoCloseScheduler.ts` pattern exactly.

Export `startRecurringScheduler()` which registers a cron job running every minute (`'* * * * *'`).

The cron callback calls `processRecurringTemplates()`:
1. `const now = new Date().toISOString();`
2. SELECT all from `recurring_templates` WHERE `is_active = 1 AND next_run <= ?` (bind `now`)
3. For each due template, call `createTicketFromTemplate(template)` in a try/catch (per-item error handling, never let one failure stop others)

`createTicketFromTemplate(template)` function:
1. Generate `const ticketId = uuidv4();`
2. Parse `template.tags` from JSON string to `string[]` — wrap in try/catch, default to `[]`
3. Filter out tag IDs that no longer exist: `SELECT id FROM tags WHERE id IN (...)` — only keep IDs that come back. Log warning if any were removed. (Per Pitfall 4 from RESEARCH.md)
4. Use `db.transaction(() => { ... })` wrapping all inserts:
   a. INSERT into `tickets` table: `(id, title, description, status, priority, category_id, requester_id, notes, solution, template_id)` — status='open', requester_id=null, notes=null, solution=null, template_id=null. Use template.title, template.description, template.priority, template.category_id.
   b. For each valid tag ID, INSERT into `ticket_tags` table: `(ticket_id, tag_id)` — match the existing ticket_tags schema.
   c. INSERT into `ticket_history`: `(id, ticket_id, user_id, field_name, old_value, new_value)` — user_id=NULL (system action, same as autoCloseScheduler), field_name='created', old_value=null, new_value='recurring:' + template.id.
   d. INSERT into `recurring_ticket_history`: `(id, template_id, ticket_id, created_at)` — id=uuidv4(), created_at=now.
   e. UPDATE `recurring_templates` SET `last_run = ?, next_run = ?, updated_at = ?` WHERE `id = ?` — last_run=now, next_run=computeNextRun(template.interval_type, template.interval_day), updated_at=now.
5. Log: `console.log('Recurring: created ticket ${ticketId} from template ${template.name}');`

`computeNextRun(intervalType, intervalDay, fromDate = new Date())` function — per RESEARCH.md Pattern 5:
- `daily`: next day at midnight — `next.setDate(next.getDate() + 1); next.setHours(0,0,0,0);`
- `weekly`: next occurrence of intervalDay (0=Sun..6=Sat), always at least 1 day ahead — `const daysUntil = (target - current + 7) % 7 || 7;`
- `monthly`: next month on `Math.min(intervalDay, daysInMonth)` at midnight. Helper: `getDaysInMonth(date) => new Date(date.getFullYear(), date.getMonth() + 1, 0).getDate()`
- Per Pitfall 1: Always compute from `new Date()` (now), never from old next_run.
- Per Pitfall 2: Clamp monthly day to last valid day of month.
- Return `next.toISOString()`

All imports: `import cron from 'node-cron'; import { v4 as uuidv4 } from 'uuid'; import { db } from '../db/connection.js';`

1. `SELECT * FROM recurring_templates ORDER BY created_at DESC`
2. For each template, fetch history: `SELECT rth.id, rth.ticket_id, rth.created_at, t.title AS ticket_title FROM recurring_ticket_history rth JOIN tickets t ON t.id = rth.ticket_id WHERE rth.template_id = ? ORDER BY rth.created_at DESC LIMIT 10`
3. Return `res.json(templates.map(t => ({ ...t, tags: JSON.parse(t.tags || '[]'), history })))` — parse tags JSON to array for frontend consumption.

**POST /** — Create template (per D-01, D-03, D-04, D-08):
1. Extract from `req.body`: `name, title, description, priority, category_id, tags, interval_type, interval_day`
2. Validate: `name` and `title` are required (400 if missing). `interval_type` must be one of `'daily','weekly','monthly'` (400 if invalid).
3. `const id = uuidv4();`
4. `const next_run = computeNextRun(interval_type, interval_day);`
5. `const tagsJson = JSON.stringify(tags || []);`
6. INSERT into `recurring_templates` with all fields. `is_active = 1`, `last_run = null`.
7. Return `res.status(201).json({ id, name, title, description, priority, category_id, tags: tags || [], interval_type, interval_day, is_active: 1, last_run: null, next_run, history: [] })`

**PUT /:id** — Update template (per D-05):
1. Extract same fields as POST from `req.body`. Also accept `is_active` (number 0 or 1) for pause/resume.
2. Check template exists: `SELECT id FROM recurring_templates WHERE id = ?` — 404 if not found.
3. Build dynamic UPDATE: only set fields that are present in req.body. Always set `updated_at = CURRENT_TIMESTAMP`.
4. If `interval_type` or `interval_day` changed, OR if `is_active` changed from 0 to 1 (resume): recompute `next_run = computeNextRun(newIntervalType, newIntervalDay)`.
5. If `tags` provided: `JSON.stringify(tags)`.
6. Execute UPDATE and return the updated template (re-SELECT after update to return full object with history).

**DELETE /:id** — Delete template (per D-05):
1. Check exists — 404 if not.
2. `DELETE FROM recurring_templates WHERE id = ?` — CASCADE deletes history rows.
3. Return `res.status(204).send()`.

**PATCH /:id/toggle** — Quick pause/resume toggle (convenience for D-05):
1. `SELECT is_active FROM recurring_templates WHERE id = ?` — 404 if not found.
2. Toggle: `new_active = current.is_active ? 0 : 1`
3. If resuming (new_active=1): recompute `next_run`.
4. UPDATE `is_active`, `next_run` (if resuming), `updated_at`.
5. Return `res.json({ id, is_active: new_active, next_run })`.
