### Toggle Interaction Flow

The Switch triggers a multi-step async flow. States must be visually distinct:

| Step | Switch State | Visual Feedback |
|------|-------------|-----------------|
| Idle (off) | unchecked | No loading indicator |
| User clicks toggle (enabling) | disabled + loading | `Loader2` spinner replaces the label icon; Switch disabled |
| Permission prompt shown | disabled | Browser native dialog — no app UI change needed |
| Permission granted → subscribing | disabled + loading | Spinner continues |
| Subscribe succeeded | checked | Toast success; status badge updates |
| Subscribe failed / permission denied | unchecked | Toast error; status badge shows blocked state if applicable |
| User clicks toggle (disabling) | disabled + loading | Spinner briefly; optimistic unchecked |
| Unsubscribe succeeded | unchecked | Toast success |

Loading pattern: use `isLoading` boolean state — disable Switch during async, show `Loader2 animate-spin` inline next to the label. Duration is short (< 1s typically) so no skeleton needed.

### Switch Row Layout

```
<div className="flex items-center justify-between">
  <div className="space-y-1">
    <Label htmlFor="push-toggle">Push-notiser</Label>
    <p className="text-sm text-muted-foreground">{statusText}</p>
  </div>
  <Switch
    id="push-toggle"
    checked={isEnabled}
    onCheckedChange={handleToggle}
    disabled={isLoading || isBlocked || isUnsupported}
  />
</div>
```

### Notification Click (OS level)

No in-app UI. The service worker `notificationclick` handler closes the notification and calls `clients.openWindow('/tickets/:id')`. This is invisible to the Settings UI component.

## Implementation Notes for Executor

1. The "Notifikationer" Collapsible section is appended after the "Backup & Export" section in Settings.tsx — it is the last section in the settings list.

2. Add `notifications: false` to the `sectionsOpen` state object alongside the existing `backup: false` entry.

3. The `isLoading` state for the toggle is separate from `backupLoading`. Name it `pushLoading`.

4. Detect `Notification.permission === 'denied'` on mount and set `isBlocked = true`. Detect `!('PushManager' in window)` and set `isUnsupported = true`. Detect iOS non-standalone via `!window.matchMedia('(display-mode: standalone)').matches && /iPhone|iPad/.test(navigator.userAgent)`.

5. The Switch is disabled (not just visually muted) when `isLoading || isBlocked || isUnsupported`. When `isBlocked` or `isUnsupported`, accompany with the appropriate instructional paragraph instead of hiding the Switch — the user needs to understand why it is disabled.

6. Permission prompt fires only inside the `onCheckedChange` handler — never in `useEffect` on mount (D-12, locked decision).
